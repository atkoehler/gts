00004 ## What is GTS
00005 Galah Testing Suite (GTS) is an automated marking, feedback and grading
00006 suite.  It is designed to best work within Galah, a web based automated
00007 testing and feedback framework but is not tied exclusively to the Galah
00008 system.  Work on Galah can be found at: https://github.com/brownhead/galah
00009 
00010 # History
00011 The Galah Testing Suite (GTS) has a mutli-step past that started out as
00012 simply a way to eliminate hand grading of programming assignments.
00013 
00014 ### Initial Premise
00015 The initial design of the testing suite was developed entirely as an
00016 automated testing and marking generation framework for CS 10 (Introduction
00017 to Computer Science) at University of California, Riverside. These sripts
00018 were written in an amalgamation of bash and C++ and as they were developed
00019 immediately prior to their use, they became quickly bloated due to the lack
00020 of design priciples beyond: it must work and must be done before the
00021 assignment is due.
00022 
00023 ### Widening the net
00024 The initial implementation was sound enough to be utilized for a few
00025 quarters.  However, expanding on the implemenation for additional
00026 assignments was impractical. Additionally, utilizing the suite in courses
00027 beyond the very first programming course was impossible.
00028 
00029 With these faults in mind.  A more clear design strategy was created to
00030 generalize many pieces utilized in the initial implementation into modules.
00031 These modules optionally combined with more specific assignment oriented
00032 tests would create a test harness to be utilized for a single assignment.
00033 Additionally, a uniform CSV output format was chosen so that outpt is
00034 easily parsed for other purposes.
00035 
00036 ## Project Goals
00037 The Galah Testing Suite has several project goals.  A few deserving more
00038 length descriptions are listed below.
00039 
00040 ### Assignment Independece
00041 A primary goal of the project is that each assignment test harness must be
00042 independent of all other test harnesses. This means that there will
00043 certainly be reproduced code when looking through multiple harnesses at
00044 once but allows the replacement or swapping of assignments without any
00045 concern for breaking assignments that are not being moved or modified.
00046 
00047 ### Easy Generation of Simple Harness
00048 A simple test harness should easily be generated or created based on
00049 generic modules. Some scripting knowledge will be required but the goal is
00050 to have a basic test harness be a short non-confounding task.
